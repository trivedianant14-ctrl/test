01 Project Overview

The NPrep QBank Prototype is a mobile-first React application that simulates a medical exam practice platform (targeted at NORCET / nursing competitive exams). The session's primary deliverable is a complete Query Management System (QMS) — allowing students to report errors or doubts on questions, track the status of their queries through a pipeline, and receive feedback once resolved.

The system was designed to work entirely within a phone-mockup UI (430px max-width), presented on desktop as a centered device frame. All QMS functionality lives inside a slide-in overlay toggled by a persistent tab button on the right edge of the phone frame — it does not interrupt the question-practice flow.

Core Goal
Let a student raise a query about any question in under 30 seconds, receive an auto-generated ticket ID, and track that query from submission to resolution — all without leaving the NPrep app.

Key User Journeys

📝
Raise a Query
Student encounters a problem → taps "Report an Error" → completes a structured 6-screen form → gets a ticket ID.
📊
Track Query Status
Student opens the Profile tab → My Queries → sees all tickets with stage badges and timestamps.
🔍
View Query Detail
Student taps a query card → sees a full timeline of how the query has progressed through the pipeline.
👍
Provide Feedback
Once resolved, student has 48 hours to thumbs-up (understood) or thumbs-down (need more help). No action = auto-close.

02 Tech Stack & Architecture

Dependencies

React 18 Vite 5 JavaScript (no TypeScript) No Tailwind No icon libraries MediaRecorder API Context API

All icons are inline SVGs. All styling is inline styles (in components) and a single index.css file with CSS custom properties. No third-party UI libraries.

State Management

A single QueryContext (React Context + useState) holds all query data globally. It is initialized from a mockQueries.js array and extended in-memory whenever a student submits a new query. No backend — all data is ephemeral in the browser session.

Architecture Decisions

DecisionRationale
Phone mockup (430px) Simulates a real mobile app without building native. All screens constrained to phone dimensions to validate mobile UX.
Overlay tracker (not a route) The query tracker must not interrupt the question-solving flow. It slides in as a full-phone overlay from an external tab button.
FormShell embedded mode The "Report an Error" form is a bottom sheet (58% height) inside the Solve screen, not a separate page — keeps question visible above.
In-memory state only Prototype stage — no backend. QueryContext stores new queries in React state so they appear immediately in the tracker.
Inline styles in components QueryTracker uses color constants (const P = '#534AB7') at the top of the file to keep theming consistent without a build-step dependency.

03 Phone Mockup Layout

On desktop (≥500px), the app renders as a phone frame (430×844px, 40px border-radius, box-shadow). On mobile the phone fills the viewport.

The Profile/Tracker toggle tab sits outside the phone div but inside a .phone-wrapper with position: relative. This solves the overflow: hidden clipping problem — children of the phone can't bleed outside it, but an absolutely-positioned sibling in the wrapper can.

// App.jsx structure <div className="phone-wrapper"> // position: relative <div className="phone"> // overflow: hidden ...question screens... {showTracker && <QueryTracker onClose={...} />} </div> <button className="tracker-tab-btn"> // position: absolute, right: -38px 👤 Profile </button> </div>

Tab Button Behaviour

  • Position: absolute; right: -38px; top: 130px — floats outside the right edge of the phone
  • Shows "Profile" label + person icon when tracker is closed; shows an X when open
  • Styled border-radius: 0 10px 10px 0 to look like a protruding tab from the phone edge
  • Hidden on mobile (display: none) — on small screens the tracker will be accessed differently

05 Voice Recording (Others Screen)

The "Others" text screen includes a VoiceRecorder component that uses the browser's MediaRecorder API. This allows students who struggle to type their doubt to record it verbally.

OR-Gate Submit Logic

Submit Condition
The Submit button activates when: text length ≥ 20 characters OR voice recording duration ≥ 10 seconds — whichever condition is met first. Both can be satisfied simultaneously.

VoiceRecorder Features

  • Single tap to start recording — live waveform animation (CSS pulsing bars)
  • Tap again to stop; audio blob is retained in component state
  • Duration tracked via onDurationChange callback prop (fires every second)
  • finalDurationRef persists the last-known duration for the submit gate
  • Playback control shown after recording is complete
  • Re-record option clears the previous blob

Props & Callbacks

<VoiceRecorder onAudio={(blob) => setVoiceBlob(blob)} onDurationChange={(secs) => setVoiceDuration(secs)} /> // Submit gate in OthersText screen: const hasText = text.trim().length >= 20 const hasVoice = voiceDuration >= 10 const canSubmit = hasText || hasVoice

06 Query Submission Success Popup

After a successful submission, screen 6 (SuccessScreen) is shown. The key addition is the generated ticket ID display — giving students a concrete reference number immediately.

Ticket ID Generation

The ID is generated in QueryContext.addQuery() using Date.now():

const id = Date.now() // e.g. 1718123456789 ticket_id: 'NP-' + id.toString(36).toUpperCase().slice(-6) // Display on SuccessScreen (last 5 digits, zero-padded): '#NP-' + String(queryId).slice(-5).padStart(5, '0')

Success Screen Elements

  • Checkmark animation — pops in with checkPop keyframe
  • Ticket ID card — dashed purple border, monospace font, "Your Ticket ID" label in uppercase
  • Tracking instruction — blue info box: "Open your profile and tap My Queries to see live status updates"
  • Notification banner — "You'll be notified when this is resolved"
  • Two CTAs — "Continue practice" (primary) and "Raise another query" (text link)

ID Threading

addQuery() was updated to return the generated ID, allowing FormShell to capture it in submittedId state and pass it as a prop to SuccessScreen.

09 Query Detail — Timeline View

Tapping any query card opens the detail view. This is modeled after Swiggy/Zomato order tracking — a vertical timeline showing each stage the query has passed through.

Query Lifecycle Stages

Query Raised Stage 0
Student submitted the query. Shows original query text in accordion dropdown.
Query Received Stage 1
Team acknowledged the query. Ticket is in the queue.
Being Worked On Stage 2
A specific agent has been assigned and is actively reviewing.
Query Resolved Stage 3
Resolution provided. Shows resolution text in accordion dropdown. 48h feedback window opens.

Stage Derivation

Stage is derived purely from the query's timeline_status field — not stored as a separate state:

const STAGE_FROM_STATUS = { raised: 0, received: 1, assigned: 2, resolved: 3, } const stage = STAGE_FROM_STATUS[query.timeline_status] ?? 0

Accordion Dropdowns

The "Query Raised" and "Query Resolved" timeline steps have expand/collapse accordions:

  • Raised accordion — reveals the student's original query text verbatim
  • Resolved accordion — reveals the resolution message written by the content team / educator
  • Arrow icon rotates 180° on open (CSS transform transition)
  • Content fades in with a max-height transition

Assigned Agent Card

At stages 2 and 3, a small agent card appears below the timeline showing:

  • Agent avatar (colored initials circle), name, and team
  • SLA indicator (e.g. "Resolves within 48 hrs")
  • 4 demo agents: Priya S. (Content QA), Rahul M. (Engineering), Sneha T. (Educator), Amit K. (Ops Triage)

Live Pulse Animation

The active stage dot pulses with a tl-pulse keyframe animation (box-shadow ripple in the primary purple) to indicate the current active state.

10 Thumbs Feedback (48-Hour Window)

Once a query reaches "Resolved" (stage 3), the ThumbsFeedback component appears below the timeline. It replaces the traditional star rating with a binary thumbs up / thumbs down.

Business Logic

Internal Ticket Lifecycle Rule
Until the student responds (👍 or 👎), the ticket remains open on the internal dashboard. The 48-hour window auto-closes it if the student takes no action, preventing stale open tickets from accumulating.

State Machine

StateConditionUI Shown
Active windowremainingMs > 0Thumbs buttons + countdown chip
UrgentremainingH ≤ 12Countdown chip turns amber/orange
Expired (no action)remainingMs ≤ 0Lock icon + "Ticket auto-closed" message
Thumbs Up chosenchoice === 'up'Celebration + "Ticket is now closed"
Thumbs Down chosenchoice === 'down'Agent card + "We're reopening your ticket"

Expiry Calculation

const resolvedTime = resolvedAt ? new Date(resolvedAt).getTime() : Date.now() const expiresAt = resolvedTime + 48 * 3600000 const remainingMs = expiresAt - Date.now() const remainingH = Math.max(0, Math.ceil(remainingMs / 3600000)) const isExpired = remainingMs <= 0

Thumbs Down Behaviour

When a student taps thumbs down, the ticket is flagged for escalation:

  • Shows the assigned agent's name and team
  • Displays: "Will reach out within 24 hours"
  • Ticket status changes to Escalated
  • Student receives notification before the call

Demo Data

Mock query NP-00004 has resolved_at set to 8 hours ago, so the live demo always shows a "40h left" countdown in the thumbs widget.
